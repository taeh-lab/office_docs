# 배포 검증 규격 (DEPLOY-AUDIT)

> **이 문서 = 배포된 라이브를 실측으로 감사하고, 회의를 소집해 다음 수를 정하는 1회 주행 규격.**
> 상위 계약: `C:\port-b\port-a\CLAUDE.md` (북극성·반-스핀·역할분담) · `toolbox\회의-규격.md` (회의) ·
> `toolbox\사고-플레이북.md` (막혔을 때).
> 이 주행은 **빌드가 아니라 감사**다. `작업방식-규격.md`의 "태스크=커밋" 레일이 아니라 여기 규격을 따른다.

## 목적 (한 줄)
**배포된 DocFinder를 앱·웹 양쪽에서 직접 만져보고, 고칠 것·이상한 것을 문서에 적고, 회의로 다음 수를 정한다.**

## 선행 조건
- `main`이 푸시되어 Vercel 배포가 끝나 있을 것. 라이브: https://office-docs.vercel.app
- **배포가 안 끝났으면 여기서 멈추고 사용자에게 보고.** 옛 버전을 감사하면 전부 헛일이다.
  판정법: 라이브 `manifest.webmanifest`의 `start_url`이 `/tool.html`이면 새 버전. `/`면 아직 옛 버전.

## 🚫 이 주행에서 하지 않는 것 (중요)
- **고치지 않는다.** 발견은 문서에만 적는다. (사용자가 "적어놓고 회의"라고 지시함)
- **배포/푸시하지 않는다.** 되돌리기 어려운 것 = 멈추고 물어보는 선.
- **로그인·결제·폼 제출 안 한다.** 라이브 프로덕션이다. 읽기와 관찰만.
  → 로그인 뒤쪽은 이 감사의 **범위 밖**. 그건 한계로 적을 것.
- **자격증명 입력 금지.** 어떤 경우에도.

## 이 문서를 읽는 에이전트에게 — 2026-07-15 세션에서 실제로 데인 것들

**전부 "재보기 전에 말해서" 틀렸다. 같은 실수를 반복하지 말 것.**

| 실수 | 실제 |
|---|---|
| 스크린샷 3번 실패 → "고장났다"고 5번 반복 | 일시적. 다시 재보니 멀쩡했고, 그 잘못된 전제로 결정을 바꿨다 |
| `grep "theme.css" tool.html` → "있다" | **주석**을 잡은 것. 브라우저는 안 불러왔다 |
| `grep "unsplash"` → "없다" | 대소문자. 실제론 `Unsplash` |
| tool.html만 재고 "pricing 링크 0개" | 소스 전수엔 8곳 더 있었다 |
| "스크롤이 잠겼다" | `scroll-behavior: smooth` 중간에 잰 것 |
| 조사 서브에이전트의 자신만만한 브리핑 | 2군데 틀림(자기가 백그라운드로 돌면서 "백그라운드 안 됨"이라 씀) |
| 내 테스트의 `/login\.html/` 정규식 → "벽 남아있음" | 내가 쓴 **주석**을 잡은 거짓 양성 |

**규칙:**
1. **숫자만 주장한다.** "느리다" 금지, "첫 페인트 4,513ms" 만.
2. **한 파일 재고 전체라고 하지 않는다.** 전수 확인 후에만 "0개".
3. **grep 결과는 주석/문자열일 수 있다.** 코드인지 확인.
4. **말한 것과 한 것이 다른지 본다.** (키 안 찍는다고 해놓고 JSON 덤프한 적 있음)
5. **못 잰 건 "확인 필요"로 적는다.** 추론을 계측인 척하지 않는다.
6. **틀린 걸 알면 즉시 정정한다.** 성공 보고 문장 안에서도.

## 검증 항목 (전부 실측. 방법은 2026-07-15에 실제로 통한 것)

도구: Browser 패널 `javascript_tool` + `computer{screenshot}` + `WebFetch`.

### A. 배포 반영 확인 (제일 먼저)
- `manifest.webmanifest` → `start_url`이 `/tool.html`인가? 아니면 **중단**.
- `/api/public-config` → `tossClientKey`(`test_ck_`), `tossSecretConfigured`, `business.*` 확인.
  ⚠️ 키 값을 로그에 통째로 덤프하지 말 것. 존재 여부만.

### B. 웹 (완전 검증 가능)
1. **공유 미리보기** — `og:image`가 절대 URL + JPEG인가. 그 URL이 **실제로 200인가**(배포 전엔 404였음).
   `og:url`·`twitter:image` 존재. **카톡 미리보기는 사용자가 직접 확인해야 함**(캐시 있음 → 카카오 디버거).
2. **h1** — index에 1개, 크기 1×1, `display`가 `none`이 **아닐 것**(none이면 스크린리더도 못 읽음).
3. **푸터** — 8개 페이지(index·pricing·terms·privacy·refund·changelog·status·signup)에 1개씩.
   사업자정보 6항목 렌더. 링크 5개 전부 **최소변 ≥24px**(WCAG 2.2 §2.5.8 AA).
4. **결제 문구** — Stripe 잔재 0곳(단 changelog v0.3의 Stripe는 **역사라 정상**).
5. **모바일 375px** — `document.scrollWidth === clientWidth`(가로 스크롤 없음).
   ⚠️ "뷰포트 밖 요소" 검사는 히어로 애니메이션을 잡는 **거짓 양성**을 낸다. 가로스크롤 여부로만 판정.
6. **성능** — 첫 방문 전송량/`loadEventEnd`. **캐시 우회 필수**(`fetch(u,{cache:'reload'})`),
   안 그러면 서비스워커가 0KB로 속인다. 폰트 4개 ≈3,063KB가 알려진 상태 — 줄었는지.
7. **콘솔 에러 / 실패한 네트워크 요청** — 전 페이지.

### C. 앱 (부분만 가능 — 한계를 정직하게 적을 것)
**실기기 TWA는 못 돌린다.** 아래는 "코드가 그렇게 분기한다"까지지 "실기기에서 그렇게 보인다"가 아니다.
1. **로직 격리 테스트** — `app-mode.js`를 fetch해서 가짜 window/location으로 실행:
   ```js
   new Function('window','location','navigator','document', src)(fakeWin, fakeLoc, fakeNav, fakeDoc)
   ```
   웹 2케이스(무변화) / 앱 비로그인·로그인(→tool.html, **login.html 아님**) / iOS(`navigator.standalone`).
2. **`.is-app` 시뮬레이션** — tool.html에서 `<html>`에 `is-app` 붙이고
   `.is-app .web-only{display:none!important}` 주입 → 결제 진입점이 `display:none`이 되는가.
   **앱에서 살아있는 `pricing.html` 링크 = 0개**여야 함(소스 전수로도 교차 확인).
   구독 해지(`#downgradeRow`)·로그인(`#amLogin`)은 **건드리면 안 됨** — 무변화 확인.
3. **설정 시트** — `LegalSheet.open('terms.html')` → 본문이 원본과 일치(≈2,289자, h2 12개).
   `'business'` → 사업자등록번호 + `[통신판매업 신고번호 준비중]`.
   ⚠️ 라이브엔 `/api/public-config`가 있으니 폴백이 아니라 **실제 값** 경로가 돈다.
4. **앱에 푸터 없음** — tool.html에 `footer.site-footer` 0개.
5. **`/.well-known/assetlinks.json`** — **404가 정상**(던스 대기 → Play Console SHA-256 미발급).
   404면 "예정된 상태"로 적을 것. 버그 아님.

### D. 눈으로 (스크린샷)
- 랜딩 · tool.html · 설정 모달 · 약관 시트 · 푸터. 데스크탑 + 375px.
- **레이아웃이 깨져 보이면 반드시 계측으로 교차 확인.** 스크린샷은 축소 렌더라
  겹쳐 보이는 거짓 양성을 낸다(2026-07-15에 CTA 겹침으로 오판한 적 있음 — 실제론 6px 여유).

## 회의 (감사 끝나면 반드시)

`toolbox\회의-규격.md`를 **그대로** 따른다. 요약이 아니라 그 파일을 읽고 따를 것.

- **등급: HEAVY** — 앱스토어 관문은 되돌리기 어렵다(패키징 후엔 재심사). 소넷 2~3명, **서로 다른 렌즈**:
  (a) 정확성·버그 (b) 단순성·과공학 (c) 리스크·되돌리기·심사
- **★ 블라인드 프레이밍 (제일 중요):** 회의엔 **문제만** 준다.
  - 준다: 목표 한 줄 / 지금 상황(사실만) / **방향성 후보 ≥5개(순위·선호 표시 없이 무표정하게)** / 3대 상시 질문
  - **절대 안 준다:** 내 결론, "3번이 맞다" 류, 내 합리화. 보면 그냥 동조한다(sycophancy).
- **적대적 스탠스**를 프롬프트에 박는다: "현재 접근이 왜 틀렸는지·뭐가 빠졌는지 찾는 게 임무. 불확실하면 회의적. 칭찬 금지."
- **결정 규칙:** 각 반론에 **명시적으로 응답**한다 — 수용하거나 "왜 아닌지" 근거를 댄다. 그냥 흘리기 금지.
  단 맹종도 아님. 소넷도 나랑 혈통이 같아 틀릴 수 있다.
- **기록:** 회의 결과 + 최종 결정 + **각 반론에 대한 대응**을 `DEPLOY-AUDIT-REPORT.md` 맨 아래 `## 회의` 절에.

**방향성 후보를 짤 때:** 감사 결과에서 나온 것 + 이미 알려진 미결(폰트 3MB / assetlinks / 케이스스터디 /
통신판매업 / 앱 심사 준비)을 섞어서 ≥5개. 순위 표시 금지.
