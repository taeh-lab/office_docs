# 크레딧 모델 — 할당량 → 재화(장부) 전환 설계

> **상태: 설계만. 구현 안 함.** (2026-07-15 사용자 결정)
> 이유: 결제 관문은 이미 완료(토스 빌링 연동)라 이 전환은 **새 증거 0**이다.
> 상위 `CLAUDE.md` 규칙 1·4(관문 밖은 골드플레이팅) + 2026-07-15 회의 만장일치("관문 밖은 지금 하지 마라")를
> 폰트 서브셋에 적용했으니, **같은 자를 여기에도 댄다.**
> **언제 하나:** 통신판매업 허가 → 토스 라이브 키 전환 시점에 같이 보는 게 자연스럽다(결제가 진짜가 되는 때).

## 목표 모델 (사용자 확정 2026-07-15)

**크레딧은 상한이 아니라 재화다.** 게임의 골드처럼 모으고 쓴다. "10만이든 1000만이든" 보유 상한은 없다.

| # | 대상 | 지급 |
|---|---|---|
| 1 | **비로그인** | **처음 접속에만 100** (1회) |
| 2 | **로그인 (무료)** | **매달 100** + 이벤트 참여 |
| 3 | **결제** | 더 많이 (플랜별) |

**단 하나의 상한:** 로그인만 하고 안 쓰는 사람이 무한정 쌓는 걸 막으려고,
**무료 사용자는 잔액 500까지만** 쌓인다. **결제 사용자는 상한 없음.**

```
잔액 = 지급 총합 − 사용 총합
지급: 첫 실행 +100 · 매달 1일 +100 · 결제 +N · 이벤트 +N
사용: 쓴 만큼 차감
보유 상한: 무료 500 / 결제 없음
```

## 왜 바꾸나 — 지금은 재화가 아니라 "데이터 요금제"다

`shared/auth.js:108`:
```js
function canSpend(email, plan, n){
  const allow = planCredits(plan);        // 플랜의 상한
  return getUsage(email) + n <= allow;    // 이번 달 사용량 + n <= 상한
}
```
- **월별 사용량을 세서 플랜 상한과 비교**한다. 매달 0으로 리셋되고, 안 쓴 건 증발한다.
- `PLANS.basic.credits = 2000`의 뜻이 **"2000까지만 쓸 수 있다"** 이지 "2000을 준다"가 아니다.
- 즉 **모으는 재화가 아니라 소진하는 한도**다.

### ★ 이 구조가 실제로 구멍을 뚫었다 (2026-07-15)
할당량 모델은 **`plan`을 알아야** 판정할 수 있고 → `plan`은 `user`에 있고 → **`user`가 없으면 예외 처리가 필요**하고
→ 그래서 메뉴에 `.acct`(로그인 전용) 같은 게 생기고 → 앱에서 푸터를 없애자 **비로그인 사용자가
약관·개인정보·사업자정보에 도달할 길이 0이 됐다**(Play 요구사항 위반). `057e633`에서 급히 막았다.

**장부 모델엔 이 연쇄가 없다.** 잔액만 보면 되니 계정도 플랜도 몰라도 된다. 막을 게 없으니 구멍도 안 난다.
> 사용자 표현: *"막을려고 하니까 막히는 거지. 걍 다 열어주고 크레딧만 바꾸면 되잖아."*

## "크레딧이 크레딧만 알면 쉬울 텐데" — 실측 결과: **아니다**

사용자 가설: *"코드를 의존성 없이 제대로 짜뒀다면, 크레딧이 아무것도 모르고 크레딧만 들고 있다면 쉬울 것."*
**틀렸다. 크레딧이 계정과 플랜을 안다.**

### 지금 시그니처 (`shared/auth.js` 81~130)
```
usageMapKey(email)          ← 계정 앎
usageMap(email)             ← 계정 앎
getUsage(email)             ← 계정 앎
planCredits(plan)           ← 플랜 앎
creditBalance(email, plan)  ← 둘 다 앎
canSpend(email, plan, n)    ← 둘 다 앎
spendCredits(email, n)      ← 계정 앎
getUsageHistory(email)      ← 계정 앎
```
→ 호출부가 **전부 `user.email` / `user.plan`을 넘긴다.** `user`가 null이면 아무것도 못 부른다.
   그래서 `chargeCredits`가 `if(!user) return true;`(무제한 통과)로 도망친다 — 이게 지금 "비로그인 무제한"의 정체.

### 폭발 반경 (좋은 소식 — 좁다)
| 위치 | 개수 |
|---|---|
| 크레딧 로직 | `shared/auth.js` 81~130, 함수 8개 (한 파일) |
| 호출부 | `tool.html` 1514·1515·1521·1533·1534·1859·1860·1873·1899 (9곳) |
| `PLANS[...]` 직접 읽기(표시용) | `tool.html` 1823·1870·1891·1933·1937·1938 (6곳) |

**파일 2개, 지점 ~15개.** 크지 않다. 다만 "두 줄"도 아니다.

## 목표 모양

### 크레딧 = 재화. 키와 숫자만 안다. 계정도 플랜도 모른다.
```js
balance(key)          // 잔액
grant(key, n)         // 지급 (더함)
spend(key, n)         // 차감. 잔액 부족이면 false
```
저장: `dwf_credits_${key}` = `{ granted: n, spent: n }` 또는 이벤트 로그.
(사용량 차트를 유지하려면 일자별 spend 기록은 남겨야 함 — 지금 `usageMap`이 하는 일.)

### 키 = 누구든 상관없음
```js
function usageKey(){
  const u = getUser();
  if (u) return u.email;
  let id = localStorage.getItem('dwf_guest_id');
  if (!id) { id = 'guest_' + crypto.randomUUID(); localStorage.setItem('dwf_guest_id', id); }
  return id;
}
```

### 지급 규칙 = 크레딧 **밖**. 여기서만 계정·플랜을 안다.
```js
grantFirstVisit()      // 비로그인 1회 +100 (플래그로 1회 보장)
grantMonthly()         // 로그인 매달 +100. 단 무료면 min(100, 500 - balance)  ← 500 상한은 여기서
grantOnPurchase(plan)  // 결제 +PLANS[plan].credits. 상한 없음
grantEvent(n)          // 이벤트
```
**상한 500을 `grant()` 안에 넣지 말 것.** 넣는 순간 크레딧이 다시 플랜을 알게 된다. 규칙 쪽에서 잘라서 넘긴다.

### 그러면 사라지는 것
- `chargeCredits`의 `if(!user) return true;` (무제한 도망) → 삭제. 잔액만 보면 됨
- `canSpend(email, plan, n)` → `spend(key, n)`
- `planCredits(plan)` → 지급 규칙 안으로
- 게스트/로그인/결제 분기 → **전부 사라짐.** 다 그냥 잔액
