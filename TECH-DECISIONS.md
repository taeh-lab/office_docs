## [2026-07-15] fail-open 소멸 — 삭제가 아니라

**증상:** 구 `auth.js:97` — `if(typeof PLANS === 'undefined') return null;` → `null` = 무제한.
`canSpend`는 `true`, `creditBalance`는 `Infinity`. **설정 누락이 조용한 과금 해제.** 크래시가 없어
감지 불가. (`null`이 정상 반환값과 구분이 안 되는 게 원인.)

**왜 그런 가드가 있었나:** `auth.js`가 `PLANS` 없는 페이지 3곳(`index`·`changelog`·`status`)에도
실렸다. 실측하니 그 3곳은 **크레딧을 0줄 쓴다** — `index.html`은 `getUser()` 1곳뿐, 나머지 둘은
**죽은 script 태그**다.

**결정:** 가드를 **지우지 않는다. 가드가 있던 함수를 파일 밖으로 옮긴다.**
`credit-rules.js`는 `tool.html`에만 실리고 거기엔 `PLANS`가 항상 있다 → 가드가 필요 없어진다 →
**fail-open이 삭제가 아니라 소멸한다.** 3개 페이지는 새 파일을 애초에 안 실으므로 깨질 표면이 0.

**그래도 순서가 깨질 경우:** `planGrant`는 `0`(fail-**closed**) + `console.error`.
`throw` 안 하는 이유는 D1("독립적으로 끝나도 앱이 안 죽게"). **`null` → `0`, 한 글자 차이가
fail-open → fail-closed.** `console.error`가 붙어서 이번엔 감지된다.

**기계 강제:** `check-credit.mjs`의 `no-fail-open` 검사가 `typeof PLANS … return null` 부활을 막는다.

## [2026-07-15] ⚠️ 검증 함정 — "값이 맞다"가 "새 코드가 돌았다"를 뜻하지 않는다

**증상:** T2 검증에서 회귀 18/18이 통과했는데, `typeof dateKey`가 `function`이었다. 삭제한 함수다.

**확인한 것:** HTTP 캐시가 **옛 `auth.js`를 서빙**하고 있었다. 로드 순서가
`auth.js`(캐시된 옛것 — `dateKey`·`usageMap`·`planCredits` 정의) → `credit-rules.js`(새것 —
같은 이름의 shim으로 **덮어씀**)라, **shim이 이겨서 값은 맞게 나왔다.** 혼합 상태였다.
`dateKey` 유령이 아니었으면 못 잡았다.

**안 통한 것 (전부 시도함):** 서비스워커 해제 · `caches` 전부 삭제 · `location.reload(true)`
(`true`는 deprecated라 무시됨) · `Ctrl+Shift+R`.
**통한 것:** **포트를 8899 → 8901로 바꿔 새 오리진을 만드는 것.**

**교훈:** 리팩터링에서 "구 이름을 shim으로 유지"하면 **캐시된 옛 코드와 새 코드가 같은 이름을
공유**해서 회귀 테스트가 통과해버린다. **새 코드만 있는 흔적**(여기선 `dateKey`의 부재)을 같이
확인해야 한다. 값만 보면 속는다.
